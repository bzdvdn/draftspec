# Prompt спецификации Draftspec

Вы создаете или обновляете одну feature specification.

## Goal

Сформируйте ясную спецификацию фичи в `.draftspec/specs/<slug>.md`, которая соответствует конституции.

Перед записью или обновлением спецификации убедитесь, что работа идет в feature-ветке для `<slug>`. Соглашение по умолчанию — `feature/<slug>`.

## Phase Contract

Inputs: смотрите Load Only.
Outputs: смотрите Output expectations.
Stop if: смотрите Stop Conditions.

## Режим работы

- Работайте ровно с одной фичей.
- Предпочитайте patch существующей спецификации вместо полного переписывания.
- Загружайте только минимум контекста, который нужен для снятия неоднозначности.
- Не уходите в планирование или реализацию.

## Load Only

- `.draftspec/constitution.md`
- текущий запрос пользователя и разговор
- минимальный контекст репозитория, необходимый для снятия неоднозначности

## Do Not Read By Default

- нерелевантные спецификации
- нерелевантные plan packages
- тяжелые implementation-части кода, если они не нужны для корректного определения scope
- contracts или data models других фич

## Stop Conditions

Остановитесь и задайте минимальный уточняющий вопрос, если:

- цель фичи неоднозначна
- запрос пытается вывести одну spec из нескольких изменений конституции без указания одного конкретного изменения или фичи
- запрос смешивает несколько фич или несвязанные изменения в одной spec
- чтобы честно удержать scope, потребовались бы несколько feature slug или несколько независимых спецификаций
- входом служит prompt-файл с generic filename без явного `name:` или `slug:`, и при этом пользователь не передал `--name` или `--slug`
- вход больше похож на URL, чем на понятное имя фичи
- критерии приемки пришлось бы выдумывать, а не выводить из запроса
- запрошенная фича выглядит конфликтующей с конституцией
- любой обязательный раздел остался бы в виде TBD или placeholder текста

Если пользователь передал `--name`, но еще не дал достаточно подробного описания фичи, не теряйте контекст запроса: запросите недостающее описание и считайте следующее пользовательское сообщение без новой slash-команды продолжением того же spec-запроса.

Не переходите к планированию или реализации, пока сама спецификация не прояснена.

Если спецификация уже существует и актуальна — сообщите об этом и не изменяйте файл.

## Инварианты

- Спецификация ДОЛЖНА соответствовать конституции.
- Держите спецификацию сфокусированной на одной фиче или одном изменении.
- Не загружайте нерелевантные артефакты других фич, чтобы компенсировать неясные требования.
- Выводите критерии приемки из запроса и реальности репозитория, а не придумывайте их.
- Используйте явные границы scope. Раздел `## Вне scope` обязателен.
- Не смешивайте языки внутри одной спецификации без сильной причины.
- Следуйте `.draftspec/templates/spec.md` при создании нового файла.
- Каждый критерий приемки ДОЛЖЕН быть в формате Given/When/Then.
- Каждый критерий приемки ДОЛЖЕН иметь стабильный ID, например `AC-001`.
- Задавайте уточняющие вопросы только когда информации критически не хватает.

## Правила разрешения входа

- `/draftspec.spec` может получать `--name <название фичи>`, опциональный `--slug <slug фичи>` и опциональный `--branch <имя ветки>`.
- Если присутствует `--name`, используйте его как каноническое название фичи.
- Если присутствует `--slug`, используйте его для пути спецификации.
- Когда `/draftspec.spec` запускается от prompt-файла, предпочитайте явные метаданные в начале файла:
  - `name: <название фичи>`
  - опционально `slug: <slug фичи>`
- Аргументы `--name` и `--slug` имеют приоритет над `name:` и `slug:` в prompt-файле.
- Если присутствует `slug:`, используйте его для пути спецификации и feature-ветки.
- Если есть `--name`, но нет `--slug`, выводите `<slug>` из `--name`.
- Если есть только `name:`, выводите `<slug>` из него.
- Переходите к basename файла только тогда, когда он достаточно конкретен для безопасного slug.
- Если пользователь явно передает `--branch <name>`, используйте это имя ветки как есть вместо default `feature/<slug>`.
- Явный `--branch` override не меняет slug спецификации, если пользователь отдельно не просит другой slug.
- Если пользователь передал только `--name`, а подробного описания еще нет, запросите описание и сохраняйте staged mode активным для следующего пользовательского сообщения без новой slash-команды.
- Если следующее пользовательское сообщение начинается с `/draftspec.`, staged mode отменяется, и новая slash-команда получает приоритет.
- Если следующее пользовательское сообщение не начинается с `/draftspec.`, считайте его продолжением staged spec-запроса.

## Правила языка

- Используйте настроенный в проекте язык документации для нового или обновляемого содержимого спецификации.
- Уважайте уже сложившееся локальное соглашение документа только если без этого сохранение существующего файла станет непоследовательным.
- Не смешивайте языки заголовков и секций внутри одной спецификации без сильной причины.

## Правила критериев приемки

- Маркеры `Given`, `When`, `Then` остаются каноническими независимо от языка документации:
  - **Given** — начальное состояние или предусловие
  - **When** — действие или событие
  - **Then** — ожидаемый наблюдаемый результат
- Критерии приемки должны быть наблюдаемыми и проверяемыми.
- Предпочитайте небольшой набор сильных критериев длинному избыточному списку.

## Output expectations

- Если окружение поддерживает создание веток, создайте или переключите `feature/<slug>` до редактирования спецификации, если только пользователь явно не передал `--branch`.
- Запишите или patch-обновите `.draftspec/specs/<slug>.md`, где `<slug>` — строчный kebab-case из названия фичи
- Кратко суммируйте цель, scope, критерии приемки и открытые вопросы

## Self-Check

- Я действительно остался в рамках одной фичи?
- Я не ушел в plan или implementation detail?
- Все критерии приемки имеют стабильный ID и Given/When/Then?
- Я не загружал нерелевантные артефакты?
