# Prompt плана Draftspec

Вы создаете или обновляете implementation plan package для одной фичи.

## Goal

Сформируйте технические planning artifacts для спецификации в `.draftspec/plans/<slug>/`.

## Phase Contract

Inputs: смотрите Load Only.
Outputs: смотрите Required outputs и Output expectations.
Stop if: смотрите Stop Conditions.

## Режим работы

- Планируйте только одну фичу.
- Предпочитайте patch существующих артефактов вместо широкого переписывания.
- Держите контекст узким и привязанным к реальности репозитория.
- Создавайте только те артефакты, которые действительно оправданы фичей.

## Load Only

- `.draftspec/constitution.md`
- `.draftspec/specs/<slug>.md`
- только те файлы кода и документации репозитория, которые нужны для планирования этой одной фичи
- если нужно читать код, предпочитайте минимальный набор файлов, достаточный для определения concrete implementation surfaces, границ и ограничений

## Do Not Read By Default

- нерелевантные спецификации
- нерелевантные plan packages
- большие области репозитория, не влияющие на эту фичу
- optional `research.md`, если неопределенность еще не подтверждена

## Stop Conditions

Остановитесь и запросите уточнение или refinement, если:

- `.draftspec/specs/<slug>.md` не существует
- спецификация слишком расплывчата для архитектурных решений, contracts или data model
- ограничения конституции конфликтуют с предполагаемым планом
- план пришлось бы вести через неясную integration- или architectural-boundary, которую не подтверждают spec и сфокусированный контекст репозитория
- работа имеет смысл только если одновременно планировать несколько feature packages

Не компенсируйте это чтением широкого нерелевантного контекста репозитория.

## Required outputs

Всегда создавайте или обновляйте:

- `.draftspec/plans/<slug>/plan.md`
- `.draftspec/plans/<slug>/data-model.md`

Создавайте только если фича действительно затрагивает:

- `.draftspec/plans/<slug>/contracts/api.md` — только если фича затрагивает API boundaries
- `.draftspec/plans/<slug>/contracts/events.md` — только если фича использует или публикует события

Создавайте `.draftspec/plans/<slug>/research.md` только если выполняется хотя бы одно из условий:

- фича зависит от внешней системы, API или зависимости, чье поведение пока неясно
- есть несколько реалистичных implementation options с существенными trade-off'ами, которые нужно сохранить
- есть неочевидный performance, security, reliability или integration risk, влияющий на планирование
- перед конкретизацией плана нужно исследовать repository constraint или архитектурную границу

## Инварианты

- План ДОЛЖЕН соответствовать конституции.
- Держите planning привязанным к текущей спецификации, а не к идеализированной архитектуре.
- Не читайте нерелевантные артефакты других фич, чтобы компенсировать нехватку ясности.
- Читайте код узко: ровно настолько, чтобы зафиксировать implementation surfaces, integration boundaries и repository constraints для этой фичи.
- Когда доступен `/.draftspec/scripts/check-plan-ready.*`, предпочитайте запускать его как phase readiness check, а не читать исходники scripts.
- Не читайте `/.draftspec/scripts/*` по умолчанию, если только не отлаживаете сами скрипты, не работаете над самим Draftspec или пользователь явно не просит разобрать script logic.
- Предпочитайте конкретные implementation decisions, а не общие советы.
- Опциональные артефакты должны оставаться опциональными; не создавайте их по привычке.
- Не создавайте `research.md` ради общего brainstorming или очевидной реализации, которую уже можно спланировать из spec и repository evidence.
- План считается достаточным только тогда, когда downstream tasks можно декомпозировать без догадок.
- План ДОЛЖЕН явно называть concrete implementation surfaces, которые предполагается изменить.
- План ДОЛЖЕН сопоставлять каждый acceptance criterion с подходом к реализации до того, как будут писаться `tasks`.
- Если автору следующей фазы пришлось бы гадать о методе, границе или пути валидации для `AC-*`, план недостаточно конкретен.
- Targeted code reading на этапе planning приветствуется, если он уменьшает downstream guesswork; широкое исследование репозитория не допускается.

## Правила языка

- Используйте настроенный в проекте язык документации для всех новых или обновляемых planning artifacts.
- Держите язык `plan.md`, `data-model.md`, `contracts/` и optional `research.md` внутренне согласованным.
- Уважайте уже сложившееся локальное соглашение документа только если без этого сохранение существующего артефакта станет непоследовательным.

## Правила трассируемости

- Следуйте структуре шаблонов `.draftspec/templates/plan.md` и `.draftspec/templates/data-model.md` при создании новых файлов.
- Data model и contracts ДОЛЖНЫ быть согласованы со спецификацией и её acceptance criteria.
- Ссылайтесь на стабильные acceptance IDs из спецификации, когда описываете acceptance-critical behavior.
- Если план фиксирует значимое implementation decision, записывайте его через стабильный decision ID, например `DEC-001`.
- Каждое значимое `DEC-*` SHOULD включать `Why`, `Affects` и `Validation`.
- Если фича вводит или меняет persisted state, переходы состояния или lifecycle rules, фиксируйте это в `data-model.md`.
- Если фича проходит через API boundary, фиксируйте request, response и error behavior в `contracts/api.md`.
- Если фича публикует или потребляет события, фиксируйте producer, consumer, payload и delivery assumptions в `contracts/events.md`.
- Не оставляйте форму сущностей, boundary IO или event payload только в prose внутри `plan.md`.
- Каждая сущность или contract entry SHOULD ссылаться на оправдывающий её `AC-*`.
- Если создается `data-model.md`, явно укажите, какие entities, invariants или lifecycle concerns этого требуют.
- Если создается `contracts/`, явно укажите, какая API-граница или event boundary этого требует.
- Если более богатые артефакты не нужны, предпочитайте не создавать их.
- Опирайтесь на реальность репозитория, а не на идеализированную архитектуру.
- Если критической информации не хватает, задавайте только минимально необходимый уточняющий вопрос.
- Формулировки вроде `обновить backend по необходимости`, `подправить логику`, `провести через систему` считайте сигналом к refinement, а не завершённым планом.

## Output expectations

- Запишите или patch-обновите артефакты плана
- Укажите, какие из опциональных артефактов созданы и почему
- Кратко опишите ключевые технические решения, которые повлияют на декомпозицию задач и реализацию
- Явно укажите риски и нерешённые вопросы, блокирующие следующие фазы

## Self-Check

- Я действительно планировал только одну фичу?
- Все опциональные артефакты действительно оправданы?
- Acceptance-critical behavior связан со стабильными ID там, где это важно?
- Из этого plan package можно писать tasks без догадок?
- Можно ли выписать первые implementation tasks без догадок о затрагиваемых surfaces и способе валидации?
