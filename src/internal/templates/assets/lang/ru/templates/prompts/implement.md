# Prompt реализации Draftspec

Вы исполняете запланированную реализацию фичи.

## Goal

Реализуйте фичу, следуя существующему списку задач и не расширяя scope.

## Phase Contract

Inputs: `.draftspec/constitution.md`, `.draftspec/plans/<slug>/tasks.md`; более глубокие артефакты — только если активная задача их требует.
Outputs: код реализации, обновлённые чекбоксы задач в `tasks.md`.
Stop if: tasks.md отсутствует, следующая задача неконкретна, scope требует выдумывать новые задачи, или все задачи уже выполнены.

## Режим работы

- Используйте `tasks.md` как execution entrypoint.
- Выполняйте минимальный безопасный scope, который разрешен запросом.
- Читайте только те артефакты и code files, которые нужны для активной задачи.
- По возможности patch-обновляйте существующие файлы вместо широкого переписывания.
- Предпочитайте readiness scripts, когда они доступны, перед чтением более глубоких артефактов.

## Load First

Всегда сначала прочитайте это перед началом реализации:

- `.draftspec/constitution.summary.md` если присутствует; иначе `.draftspec/constitution.md`
- `.draftspec/plans/<slug>/tasks.md`

## Load If Present

Читайте это только когда этого требует активная задача:

- `.draftspec/specs/<slug>/summary.md` если присутствует; иначе `.draftspec/specs/<slug>/spec.md` — когда неясны intent задачи или границы приемки
- `.draftspec/plans/<slug>/plan.md`, когда нужна архитектурная стратегия или порядок работ
- `.draftspec/plans/<slug>/data-model.md`, когда важны форма данных, инварианты или поведение жизненного цикла
- `.draftspec/plans/<slug>/contracts/`, когда затрагиваются API, события или интеграционные контракты
- `.draftspec/plans/<slug>/research.md` только если файл существует и текущая задача действительно от него зависит
- только те code files, которые нужны для активных задач
- начинайте с implementation surfaces, уже зафиксированных в плане или задачах, и только потом расширяйте code inspection

Не считайте, что `research.md` обязан существовать; используйте его только когда активная задача зависит от сохраненной неопределенности, внешней зависимости или зафиксированного trade-off.

## Do Not Read By Default

- нерелевантные контракты
- весь код репозитория, если реально нужны только несколько файлов
- большую историческую переписку, если нет блокера

## Stop Conditions

Остановитесь и запросите refinement, если:

- `.draftspec/plans/<slug>/tasks.md` отсутствует
- следующая задача недостаточно конкретна для безопасной реализации
- текущей задаче не хватает spec, plan, data model или contracts
- план конфликтует с конституцией
- реализация требует scope за пределами текущего task list
- выбранная работа заставляет менять другой feature package или slug, который не входит в текущий task scope
- следующий безопасный шаг потребовал бы придумывать новые задачи или acceptance coverage

Если все задачи в `tasks.md` уже отмечены выполненными — сообщите об этом и не продолжайте реализацию.

Не расширяйте scope, чтобы обойти эти проблемы.

## Правила scope

- Поведение по умолчанию: если пользователь явно не ограничил scope, выполняйте только первую незавершенную фазу или минимальный непрерывный кластер незавершенных задач, который нужен для forward progress.
- Scoped behavior: если пользователь явно передает `--phase <номер>`, выполняйте только эту фазу.
- Scoped behavior: если пользователь явно передает `--tasks <список-task-id>`, выполняйте только эти task IDs.
- Не принимайте `--phase` и `--tasks` одновременно в одном запуске.
- В scoped mode сохраняйте порядок из `tasks.md`, а не придумывайте новый порядок из текста запроса.
- Если выбранная фаза или task IDs отсутствуют в `tasks.md`, остановитесь и запросите refinement.
- Если scoped execution перескакивает через незавершенную более раннюю работу, предупредите о риске порядка, но не расширяйте scope молча.

## Дисциплина чтения

**Каждый файл читается один раз за сессию.** Не перечитывайте файл, который уже загружен, если только вы сами его не редактировали и не нуждаетесь в проверке конкретной детали своей правки. После собственной правки вы уже знаете что изменилось — перечитывать «для уверенности» расточительно.

**Протокол начала сессии** — перед выполнением первой задачи:
1. Запустите `.draftspec/scripts/check-implement-ready.*`, если доступен.
2. Прочитайте `tasks.md` и определите все задачи в текущем scope.
3. Соберите `Touches:` surfaces из каждой задачи в scope; если в `tasks.md` есть `## Surface Map`, читайте его вместо сканирования отдельных строк задач.
4. Прочитайте эти surfaces разом, одним batch, до начала любой реализации.
5. Выполняйте задачи из этого предзагруженного контекста без повторного открытия файлов между задачами.

**Перечитывайте только когда:**
- Файл был изменён извне (например, пользователем в середине сессии).
- Конкретная задача явно требует проверить текущее состояние файла после того, как предыдущая задача изменила его неочевидным образом.

**Не перечитывайте между задачами** чтобы «обновить контекст» или «убедиться» — используйте то, что загружено в начале сессии.

## Инварианты

- Реализуйте только незавершенные задачи из `tasks.md`.
- Соблюдайте порядок и фазовую структуру в `tasks.md`.
- Не перепроектируйте и не перепланируйте фичу молча во время реализации.
- Если вы изменили файл, не указанный в `Touches:` активной задачи, остановитесь и объясните почему до того, как продолжить. Незаявленные изменения поверхностей — это нарушение scope, а не улучшение.
- Не читайте по умолчанию нерелевантные артефакты других фич или широкие области репозитория.
- Не перечитывайте файлы между задачами; опирайтесь на batch-чтение из начала сессии и историю собственных правок.
- Когда доступен `/.draftspec/scripts/check-implement-ready.*`, предпочитайте запускать его как phase readiness check, а не читать исходники scripts.
- Не читайте `/.draftspec/scripts/*` по умолчанию, если только не отлаживаете сами скрипты, не работаете над самим Draftspec или пользователь явно не просит разобрать script logic.
- Если задачу нельзя безопасно реализовать из текущих артефактов, остановитесь и запросите refinement.
- Если для продолжения нужно сделать неочевидное допущение (форма API, формат данных, обработка ошибок), зафиксируйте его как `[ASSUMPTION: ...]` в progress output до того, как действовать. Если допущение существенно влияет на acceptance scope — остановитесь и спросите.
- Отмечайте завершенные задачи в `tasks.md`.
- Держите runtime updates короткими и привязанными к текущей фазе и task IDs.
- Не нарушайте конституцию.
- Оставляйте фичу в таком состоянии, чтобы следующая verify-проверка могла понять, что изменилось, что осталось и почему задача считается завершенной, без догадок.
- Не перепланируйте фичу, не выдавайте verify verdict и не закрывайте молча соседние задачи, которые не входили в активный execution scope.

## Правила прогресса

- Всегда явно сообщайте, какая фаза сейчас находится в работе, если активная реализация пересекает границу фаз.
- Когда фаза становится завершенной внутри текущего execution scope, выдавайте короткий phase-completion update с названием фазы и завершенными task IDs.
- Такие runtime progress updates должны быть на настроенном в проекте языке общения с агентом, чтобы пользователь не получал полностью английские phase-status сообщения в неанглийском workflow.
- Используйте короткие progress lines в стабильном формате:
  - `[T1.1] started`
  - `[T1.1] done`
  - `[T1.1] blocked: <reason>`
  - `[Phase 1] done: T1.1, T1.2`
- Загружайте более глубокие артефакты только когда этого требует текущая задача.

## Правила handoff

- Перед тем как отмечать задачу выполненной, убедитесь, что наблюдаемый результат из текста задачи действительно достигнут. Явно укажите: какой файл изменился, что добавлено или изменено и в чём состоит observable proof. Не отмечайте done, если единственное доказательство — «код выглядит правильно».
- Если задача ссылается на `AC-*`, держите реализацию в рамках этой acceptance scope и не расширяйте поведение молча.
- Когда активный scope завершен, оставляйте достаточно evidence для следующей фазы:
  - обновленные чекбоксы в `tasks.md`
  - краткое summary того, что изменилось
  - ясные blockers или оставшиеся открытые задачи
- Если по ходу реализации выясняется, что task text, acceptance coverage или план неверны, остановитесь и верните workflow на refinement `tasks` или `plan`, а не чините process contract молча в коде.

## Правила языка

- Следуйте предпочтительному языку комментариев в коде, который зафиксирован в `.draftspec/draftspec.yaml` и `.draftspec/constitution.md`.
- Когда добавляете или редактируете комментарии в коде, держите их на настроенном языке комментариев, если только в самом файле уже нет устойчивого локального соглашения, которое нужно сохранить.
- Не вводите смешанные по языку комментарии в одной локальной области кода без сильной причины.
- Если плана или задач недостаточно, остановитесь и запросите refinement вместо того, чтобы выдумывать широкий новый scope.

## Output expectations

- Выполните работу; обновите чекбоксы в `tasks.md`; сообщайте phase progress в формате `[T1.1] done`
- Явно укажите, какие acceptance criteria теперь покрыты; не заявляйте покрытие, которое не реализовано
- Завершайте разговор summary block: `Slug`, `Status`, `Artifacts`, `Blockers`, `Next command`
- Когда scope готов к проверке: `Следующая команда: /draftspec.verify <slug>`

## Self-Check

- Я выполнил только тот scope, который разрешен `tasks.md` и запросом?
- Я обновил выполненные задачи и coverage по acceptance criteria?
- Фаза `verify` поймет, что изменилось и что осталось, без повторного чтения всего репозитория?
