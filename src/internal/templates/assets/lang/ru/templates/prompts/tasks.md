# Prompt задач Draftspec

Вы создаете или обновляете `.draftspec/plans/<slug>/tasks.md`.

## Goal

Разбейте согласованный план на исполнимые implementation tasks.

## Phase Contract

Inputs: `.draftspec/constitution.md`, `.draftspec/plans/<slug>/plan.md`; опционально spec, data-model, contracts, если декомпозиция их требует.
Outputs: `.draftspec/plans/<slug>/tasks.md` с фазовым списком задач и секцией Acceptance Coverage.
Stop if: plan.md отсутствует, план слишком расплывчат, или хотя бы один AC нельзя привязать к выполнимой задаче.

## Режим работы

- Декомпозируйте только один согласованный plan package.
- Используйте `plan.md` как entrypoint и углубляйтесь только по необходимости.
- Стройте минимальный task list, который все еще безопасно покрывает фичу.
- Предпочитайте явную последовательность расплывчатым umbrella tasks.

## Load First

Всегда сначала прочитайте:

- `.draftspec/constitution.md`
- `.draftspec/plans/<slug>/plan.md`

## Load Only If Needed

Читайте это только если текущая декомпозиция действительно этого требует:

- `.draftspec/specs/<slug>.md`, когда непонятны intent задачи, scope или границы приемки
- `.draftspec/plans/<slug>/data-model.md`, когда декомпозиция зависит от сущностей, инвариантов или жизненного цикла
- `.draftspec/plans/<slug>/contracts/`, когда задачи затрагивают API, события или границы интеграций
- `.draftspec/plans/<slug>/research.md` только если файл существует и влияет на последовательность реализации или риски
- минимальный набор implementation files, нужный для подтверждения границ задач, порядка или конкретных outcome

Не считайте, что `research.md` обязан существовать; используйте его только когда план явно зависит от сохраненной неопределенности, внешней зависимости или зафиксированного trade-off.

## Do Not Read By Default

- нерелевантные спецификации
- нерелевантные plan packages
- implementation-файлы, не нужные для декомпозиции работы
- широкую историю репозитория

## Stop Conditions

Остановитесь и запросите refinement, если:

- `.draftspec/plans/<slug>/plan.md` отсутствует
- задачи получаются расплывчатыми из-за недостаточно конкретного плана
- текущей декомпозиции не хватает spec, data model, contracts или research
- конституция блокирует предлагаемую декомпозицию
- декомпозиция расползается на несколько feature slug или несвязанные change sets
- один или несколько acceptance criteria нельзя сопоставить с исполнимой работой без догадок

Не переходите вперед к реализации.

## Инварианты

- Задачи ДОЛЖНЫ соответствовать плану и конституции.
- Используйте `plan.md` как entrypoint для декомпозиции.
- Не читайте нерелевантные артефакты других фич, чтобы компенсировать слабый план.
- Читайте implementation code только если без этого task list останется расплывчатым; предпочитайте узкий срез файлов, а не широкое исследование.
- Когда доступен `/.draftspec/scripts/check-tasks-ready.*`, предпочитайте запускать его как phase readiness check, а не читать исходники scripts.
- Не читайте `/.draftspec/scripts/*` по умолчанию, если только не отлаживаете сами скрипты, не работаете над самим Draftspec или пользователь явно не просит разобрать script logic.
- Task list должен быть исполним по порядку.
- Каждый критерий приемки должен быть покрыт хотя бы одной задачей.
- Предпочитайте конкретные, проверяемые и implementation-oriented задачи.
- Добавляйте шаги валидации и синхронизации документации только когда они действительно нужны.
- Не создавайте расплывчатые umbrella tasks.
- Task list должен быть читаемым и для implementation-агента, и для reviewer без дополнительной интерпретации.
- Targeted code reading во время декомпозиции полезен, если он уменьшает повторное чтение на этапе implementation.
- Не начинайте implementation work, не редактируйте исходный код и не заявляйте, что задачи уже выполнены, на фазе tasks.

## Правила языка

- Используйте настроенный в проекте язык документации для нового или обновляемого содержимого задач.
- Уважайте уже сложившееся локальное соглашение task-документа только если без этого существующий файл станет непоследовательным.
- Не смешивайте языки внутри одного task list без сильной причины.
- Загружайте более глубокие артефакты только когда это действительно нужно текущей декомпозиции.

## Правила формата задач

- Следуйте структуре шаблона `.draftspec/templates/tasks.md`: группируйте задачи по упорядоченным фазам (`## Фаза N: Название`).
- Каждая задача ДОЛЖНА иметь phase-scoped task ID в формате `T<phase>.<index>`.
- Каждая задача ДОЛЖНА иметь формат: `- [ ] T<phase>.<index> <глагол действия> — <конкретный измеримый результат>`
- Каждая задача SHOULD по возможности ссылаться на 1-2 стабильных ID (`AC-*`, `RQ-*`, `DEC-*`).
- Если это реально уменьшает downstream guesswork, добавляйте короткий `Touches:` hint к цели фазы или строке задачи, называющий конкретные surfaces, которые с высокой вероятностью будут затронуты. Держите его компактным.
- Задачи в совокупности ДОЛЖНЫ покрывать все acceptance criteria из спецификации. Если критерий не покрыт ни одной задачей — это blocker.
- Секция `## Покрытие критериев приемки` ДОЛЖНА содержать хотя бы одну явную строку покрытия для каждого критерия приемки.
- Строки покрытия SHOULD ссылаться на стабильные acceptance IDs и task IDs, например `AC-001 -> T1.1, T2.1`.
- Для новых task lists task IDs обязательны.
- Если существующий task list без task IDs существенно редактируется, нормализуйте его к ID-based формату.

## Правила качества содержимого

- У каждой фазы должна быть короткая цель, объясняющая, зачем эта фаза существует.
- Предпочитайте несколько конкретных задач с измеримым результатом множеству мелких bookkeeping items.
- Используйте action verbs, привязанные к наблюдаемой работе: implement, add, migrate, validate, remove, backfill, document.
- Отделяйте foundational setup от core behavior и отдельно выносите proof/validation вместо смешивания его с общей реализацией.
- Если задаче полезен hint по surface, предпочитайте компактную пометку вида `Touches: auth flow, session store`, а не спекулятивные списки файлов поштучно.
- Если задача нужна только для доказательства поведения, говорите об этом явно, а не прячьте это внутри большой implementation-задачи.
- Если фаза не нужна этой фиче, пропустите её или явно укажите, что она осознанно не требуется, вместо generic filler tasks.
- Текст задачи должен делать ожидаемый результат очевидным для reviewer без постоянного возврата в plan.
- Negative examples: избегайте `misc`, `cleanup as needed`, `провести всё до конца`, `финальная полировка` и других формулировок, которые прячут результат за общим глаголом.
- Избегайте формулировок вроде `misc`, `cleanup as needed`, `доделать по месту`, `финальная полировка` и других vague umbrella phrases.

## Output expectations

- Запишите или patch-обновите `.draftspec/plans/<slug>/tasks.md`
- Убедитесь, что задачи можно выполнять по порядку
- Явно укажите блокеры или недостающие входы, если декомпозиция пока невозможна
- Когда упоминаете созданные или обновленные файлы в разговоре, указывайте их точные project-relative пути, а не только короткие имена файлов
- Завершайте разговор коротким стабильным summary block с полями `Slug`, `Status`, `Artifacts`, `Blockers` и `Next command`, если такой handoff действительно безопасен
- Когда декомпозиция завершена и можно переходить к реализации, завершайте разговорную сводку строкой `Следующая команда: /draftspec.implement <slug>`
- Если задачи заблокированы или еще требуют refinement, говорите об этом прямо вместо подсказки `/draftspec.implement`

## Self-Check

- Я действительно начинал декомпозицию с `plan.md`?
- У каждой задачи есть стабильный task ID и измеримый результат?
- Все acceptance criteria покрыты явно?
- Я не читал implementation files без реальной необходимости?
- Если я читал код, то только минимальный срез, который нужен, чтобы не оставить задачи расплывчатыми?
- Другой разработчик сможет выполнить эти задачи по порядку без догадок о том, что считается `done`?
