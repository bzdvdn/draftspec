# Prompt задач Draftspec

Вы создаете или обновляете `.draftspec/plans/<slug>/tasks.md`.

## Goal

Разбейте согласованный план на исполнимые implementation tasks.

## Режим работы

- Декомпозируйте только один согласованный plan package.
- Используйте `plan.md` как entrypoint и углубляйтесь только по необходимости.
- Стройте минимальный task list, который все еще безопасно покрывает фичу.
- Предпочитайте явную последовательность расплывчатым umbrella tasks.

## Load First

Всегда сначала прочитайте:

- `.draftspec/constitution.md`
- `.draftspec/plans/<slug>/plan.md`

## Load Only If Needed

Читайте это только если текущая декомпозиция действительно этого требует:

- `.draftspec/specs/<slug>.md`, когда непонятны intent задачи, scope или границы приемки
- `.draftspec/plans/<slug>/data-model.md`, когда декомпозиция зависит от сущностей, инвариантов или жизненного цикла
- `.draftspec/plans/<slug>/contracts/`, когда задачи затрагивают API, события или границы интеграций
- `.draftspec/plans/<slug>/research.md` только если файл существует и влияет на последовательность реализации или риски

## Do Not Read By Default

- нерелевантные спецификации
- нерелевантные plan packages
- implementation-файлы, не нужные для декомпозиции работы
- широкую историю репозитория

## Stop Conditions

Остановитесь и запросите refinement, если:

- `.draftspec/plans/<slug>/plan.md` отсутствует
- задачи получаются расплывчатыми из-за недостаточно конкретного плана
- текущей декомпозиции не хватает spec, data model, contracts или research
- конституция блокирует предлагаемую декомпозицию

Не переходите вперед к реализации.

## Инварианты

- Задачи ДОЛЖНЫ соответствовать плану и конституции.
- Используйте `plan.md` как entrypoint для декомпозиции.
- Не читайте нерелевантные артефакты других фич, чтобы компенсировать слабый план.
- Когда helper scripts для readiness или cheap validation доступны и подходят к текущей фазе, предпочитайте запускать их, а не читать их исходники.
- Не читайте `/.draftspec/scripts/*` по умолчанию, если только не отлаживаете сами скрипты, не работаете над самим Draftspec или пользователь явно не просит разобрать script logic.
- Task list должен быть исполним по порядку.
- Каждый критерий приемки должен быть покрыт хотя бы одной задачей.
- Предпочитайте конкретные, проверяемые и implementation-oriented задачи.
- Добавляйте шаги валидации и синхронизации документации только когда они действительно нужны.
- Не создавайте расплывчатые umbrella tasks.

## Правила языка

- Используйте настроенный в проекте язык документации для нового или обновляемого содержимого задач.
- Уважайте уже сложившееся локальное соглашение task-документа только если без этого существующий файл станет непоследовательным.
- Не смешивайте языки внутри одного task list без сильной причины.
- Загружайте более глубокие артефакты только когда это действительно нужно текущей декомпозиции.

## Правила формата задач

- Следуйте структуре шаблона `.draftspec/templates/tasks.md`: группируйте задачи по упорядоченным фазам (`## Фаза N: Название`).
- Каждая задача ДОЛЖНА иметь phase-scoped task ID в формате `T<phase>.<index>`.
- Каждая задача ДОЛЖНА иметь формат: `- [ ] T<phase>.<index> <глагол действия> — <конкретный измеримый результат>`
- Задачи в совокупности ДОЛЖНЫ покрывать все acceptance criteria из спецификации. Если критерий не покрыт ни одной задачей — это blocker.
- Секция `## Покрытие критериев приемки` ДОЛЖНА содержать хотя бы одну явную строку покрытия для каждого критерия приемки.
- Строки покрытия SHOULD ссылаться на стабильные acceptance IDs и task IDs, например `AC-001 -> T1.1, T2.1`.
- Для новых task lists task IDs обязательны.
- Если существующий task list без task IDs существенно редактируется, нормализуйте его к ID-based формату.

## Output expectations

- Запишите или patch-обновите `.draftspec/plans/<slug>/tasks.md`
- Убедитесь, что задачи можно выполнять по порядку
- Явно укажите блокеры или недостающие входы, если декомпозиция пока невозможна

## Self-Check

- Я действительно начинал декомпозицию с `plan.md`?
- У каждой задачи есть стабильный task ID и измеримый результат?
- Все acceptance criteria покрыты явно?
- Я не читал implementation files без реальной необходимости?
